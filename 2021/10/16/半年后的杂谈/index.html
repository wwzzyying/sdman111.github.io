半年后的杂谈

2021-10-16

距离上次发文章已经快半年了，这半年一次写技术博客的冲动都没有，倒也不是没东西可写，只是单纯懒了哈哈哈哈，难得周末有空就写一下杂谈吧！

刚开始实习时

因为一直都是写 OC 和 Swift 嘛，我以为大公司维护一个 iOS 项目也是只需要写业务就行，没想到进去的第一天就得知小组原来是分为基础组和业务组，我归到基础组来进行实习，另一位同学归到业务组。直观上区别就是我平时的工作内容就是使用 ruby 来写脚本，业务的同学使用 OC/Swift 来写 Demo；之所以使用 ruby 是因为 cocoapods 这个包管理工具就是用 ruby 写的，并且提供了许多内置的类似 xcodeproj 的内置库来修改项目，可以直接修改项目构建设置等等。

前一两个月因为实习生不能看项目源码，所以只能做一下围绕提高项目开发效率或者分析构建产物的工具，旨在优化项目、定位可能出现的问题；

第一个实践

第一个接触的实践就是使用 ruby 写一个分析构建产物 LinkMap 的脚本，主要目的是生成每个类与链接进最终产物的该类的函数的索引，然后还实现包体积分析、包含的 bundle 包体积、每个通过 pod 引入的组件链接进最终产物的体积、每个 .o 目标文件链接进最终产物的符号体积、通过类和函数的索引来搜索某个函数所在的目标文件和所在类、寻找被分类重写的方法并且输出所在位置、区分 development pod 和 remote pod（需要 hook 一下 post_install）。

总体是一个 command line tool，但是毕竟是第一次写命令行程序，写得还是太生硬了，比如命令行可以通过 >> 将执行结果进行输出，而我还硬生生地加了个格式化保存结果的 option …

总而言之第一个命令行工具是一个不及格的工具，还是对命令行的使用太少导致太多无用工作的产生。

主要的实现效果大概看看吧哈哈哈哈，用是能用，但是还是有问题，比如类里面的 C 函数好像是没识别到的

其实相当于 linkmap_ios 开源工具的一个扩展

第二个实践

第二个接触的实践是写一个头文件整理的命令行工具，也就是将 #import "A.h", #import "A/A.h" 或者 #import <A.h> 这种不规范引入整理成 #import <A/A.h> 。这个工具也是我觉得第一个让我满意的命令行工具，因为不但没有了前一次命令行工具难用，并且使用了 Rainbow 库的使用，增强了脚本执行结果的可视效果。

通过学习美团的思琦大佬基于 Swift 写的命令行工具 ImportSanitizer 使用 ruby 进行扩展，扩展了头文件分组件排序、考虑宏定义的特殊情况、标准化之后的头文件去重，通过扫描项目 pod install 之后生成的 Pods 目录来生成每个 头文件 -> 标准引入名 的映射表，后面直接正则扫描源文件然后替换就行。

不过写这个工具的时候并没有项目权限，导致全部都是基于 demo 的测试，不能覆盖到项目的其他情况，所以也并不完善，主要是宏这个考虑的情况可以有太多了，落地到公司项目估计还需要修改

第三个实践

这个实践是关于找出每个源文件中无用头文件的，不过这个比较难，需要往编译器那方面去弄，至今也没弄出来，之前了解到有一个点就是可以试试 Clang 的抽象语法树 AST 来尝试；还有一点是通过 dump 一下目标文件来得到引用到的头文件，再结合其他操作间接达到寻找无用头文件的效果。

可以看看下面就是 dump 某个 .o 后筛选的效果，这部分头文件是确定引用到的，但尝试的途中遇到了一个问题，就是当某个头文件是某个类的分类时，并不能得到这个头文件是否被引用的结果，我猜测可能与分类的加载时间有关。

第四个实践

也是我现在一直在公司维护的，叫二进制和源码的切换方案，其中涉及到自动打包工具的编写，参考了 cocoapods-packager 这个插件，将该插件转变为普通脚本，然后根据需求定制化一下，并且接入到 pod install 中，通过本地维护的一个 plist 表，在 post_install 期间校验哪些第三方库需要预编译然后进行编译并缓存到指定目录。

这样的好处是一次预编译就可以省掉后面对这些组件的再次编译，缩短构建耗时，把这部分组件参与的构建环节缩短到链接部分，仅参与最终产物的链接即可。

但是缩短的耗时并不一定明显，这部分耗时会随着第三方组件在全部组件中的占比降低而降低，如果你的项目基本都是第三方组件或者公司内部不需要频繁修改的组件，建议直接二进制化，那样编译耗时降低十分明显。

目前

在维护二进制源码方案的同时也在寻找缩短构建耗时的方法，比如 header map 的尝试，参考美团的cocoapods-hmap-prebuilt 以及 58 同城的cocoapods-wbhmap，我也在各位巨人的肩膀上进行了尝试，结合 hmap 中 json 转为 hmap 这个功能的 json 格式以及 cocoapods-hmap ，大概弄了个东西ruby-hmap-prebuilt 能够直接接入 post_install ，大概设想也可以用在构建机上面，不再需要依赖插件的安装；但是在实践下来直接替换全部 Header Search Path 会出现编译问题，需要对一个个组件的可用性进行筛选，本地编译过了还要放在构建机上尝试，但我在构建机上尝试还失败了，弄得我没脾气了，反正这一块很多坑需要踩，并不是如 58 那一篇文章所说的直接替换然后关闭原生的 Header Map 这么简单的，需要考虑自身项目的特殊性，我发现很多头文件有问题的都是 C 相关的以及 RN 组件相关的。

其实二进制预编译那块，如果调试第三方组件源码的频率很低的话基本可以按照网上出现最多的方法来接入，就直接打包好之后按组件划分然后构建每个组件的 podspec 放到专门的仓库中，拉取就直接拉取 .a 以及头文件就好了，然后弄一个环境变量在需要切换到源码的时候 pod install 切换一下，按之前跟大佬的讨论的说法，这部分都丢到公司内网中，切换速度是很快的。

如果公司内部的组件全部 pod lint 并且能够单独编译通过的话，也是一笔隐藏的可以缩短构建耗时的关键。

大概也就是这些了

感觉围绕 ruby 来管理项目还是很爽的，存在许多不可能和可能。当然了，如果要往进一步的缩短编译耗时去做，估计得好好学习 LLVM 相关知识了，毕竟看一下 19 年的 iOS微信编译速度优化分享 ，发现自己学习到的仅仅是冰山一角，好好学习吧哈哈哈哈。

这段时间从实习到答辩到参与小组工作，感觉慢慢适应了，当然菜还是得多学习，把不要把项目搞崩作为第一大原则哈哈哈哈。
